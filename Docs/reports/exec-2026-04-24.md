## What Shipped This Period

- **Strawberry bug fixed** — a user reported "5 pieces = 750g = 240 cal" (real: ~60g ≈ 19 cal, a **4× overcount**). The root cause was a fallback that synthesized `piece = food.servingSize` whenever the food wasn't in a 20-entry produce dictionary. Fix gates the unit behind a trusted source (new `pieceSizeG` override column or the dict), and the first 11 berry/grape foods are backfilled with USDA-measured piece weights.
- **Systemic unit-layer hardening** — same anti-pattern appeared at **117 sites** in `ServingUnit.swift` potentially affecting **~1,242 of 2,511 foods** (49%). Added nullable per-food override columns (`pieceSizeG`/`cupSizeG`/`tbspSizeG`/`scoopSizeG`/`bowlSizeG`) via migration v34, post-hoc reconciliation so overrides can't be drowned out by upstream branches, and UI now prefixes estimated gram figures with "≈" so users see the difference between measured and guessed.
- **Online food fallback stops discarding USDA data** — `USDAFoodService` was throwing away the `foodPortions` array from the API response (authoritative per-unit gram weights) and hardcoding 100g. Now parsed and propagated into `Food.pieceSizeG`/`cupSizeG`/`tbspSizeG`, plus OpenFoodFacts `piecesPerServing` flows through to `pieceSizeG` instead of being dropped at the adapter.
- **AI chat meal logging unified** — user logged "breakfast avocado toast and coffee" and reported "only avocado toast was stored." Root cause: two divergent log paths (AI chat's `QuickAddView` defaulted to aggregated single-entry, Food tab's `ComboLogSheet` always expanded). Extracted shared `FoodLogViewModel.logRecipeItems` helper; AI-chat multi-item logs now default to expanded so the diary rows match the conversation.
- **Combos & Recipes editable from the list** — edit was reachable only by swiping left (invisible to users); added a visible pencil icon and row-tap-to-edit.

## Risks & Blockers

- **Bulk enrichment is still a massive todo** — 11 of 2,511 foods backfilled this cycle. The remaining ~2,000 need a scripted USDA/OpenFoodFacts pass to populate piece/cup/tbsp overrides. Until then, the gate means users get `g`/`cup`/`serving` for most foods (safe but less convenient) instead of `piece`.
- **Indian regional long tail** — USDA is thin on dosa/pakora/gulab jamun/laddu variants. Roughly 600 foods will need nutritionist review or a grounded LLM enrichment pass. Admin input needed on how to prioritize.
- **Schema-level 2× overcounts** still present — `Idli (2 pieces): 80g` and similar entries encode multiple pieces in a single serving_size, and the piece-unit math doesn't know to halve it. Separate data-cleanup pass required.
- **Bulk enrichment still ~2,000 foods out** — until backfilled, the piece-gate means users get `g`/`cup`/`serving` for most foods (safe, less convenient). A scripted USDA/OFF pass is the highest-leverage next step.
